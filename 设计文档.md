OpenCompletion目标是实现一个类似 Copilot 的 VS Code 智能补全插件，但后端模型来自本地 Ollama 或任意兼容 OpenAI API 协议的服务。本指南从技术选型、整体架构、核心能力实现方式、代码规范与工程规范几个层面给出一套可落地的设计。

一、整体技术栈选择

插件本体基于 VS Code Extension API，语言使用 TypeScript。它与 VS Code 的编辑器事件、补全系统、命令系统深度集成，TypeScript 能获得完整类型定义和官方示例支持。

网络通信层使用原生 fetch（Node 18+）或 axios，用于调用兼容 OpenAI 协议的接口，如 /v1/chat/completions 或 /v1/completions。为了兼容 Ollama，需要允许自定义 baseURL，例如 [http://localhost:11434/v1。](http://localhost:11434/v1。)

配置管理依赖 VS Code 的 workspace.getConfiguration()，用于读取 API_URL、API_KEY、上下文范围等设置。

上下文分析与代码解析优先使用轻量策略（文本切片 + 启发式规则），复杂语言理解不在插件内做 AST 级重解析，避免性能问题。必要时可针对主流语言引入 tree-sitter，但默认不强依赖。

插件在右下角显示图标，点击图标可以打开设置页面，设置页面可以设置 API_URL、API_KEY、上下文范围等设置。并具有功能开启关闭开关。

日志与调试建议使用 outputChannel（window.createOutputChannel），不要直接 console.log。

二、插件总体架构

建议采用分层结构，而不是把逻辑全部写在 extension.ts。

典型目录结构：

src/

* extension.ts            插件入口，生命周期管理
* config/

  * settings.ts           统一读取与监听配置
* api/

  * llmClient.ts          与 OpenAI 兼容接口交互
  * promptBuilder.ts      构造补全/编辑 Prompt
* completion/

  * inlineProvider.ts     行内补全核心（Copilot 风格）
  * contextCollector.ts   上下文收集逻辑
* edit/

  * docCommand.ts         文档改写/续写命令
* utils/

  * debounce.ts
  * tokenizer.ts（可选）
  * pathUtils.ts

核心思想是：编辑器事件层、上下文构造层、模型交互层、展示层解耦。任何模型替换只影响 llmClient，不影响 VS Code 集成逻辑。

三、与模型交互的统一抽象

定义一个 LLMClient 接口，而不是在各处直接写 HTTP 请求。

接口示例语义：

* completeCode(context: CompletionContext): Promise<string>
* rewriteDocument(context: EditContext): Promise<string>

内部封装：

1. 读取 API_URL 和 API_KEY。
2. 构造标准 OpenAI 格式请求：
   model: 用户可配置
   messages: system + user
   temperature、max_tokens 可给默认值。

对 Ollama 兼容的关键是只使用通用字段，不依赖 OpenAI 专有扩展。错误处理统一在这一层做，向上只抛业务错误。

四、Copilot 风格补全的实现方式

关键 API 是 registerInlineCompletionItemProvider。这是实现“虚化灰字补全”的核心。

流程如下：

1. 监听编辑器触发补全事件（系统会自动在合适时机调用 provider）。
2. 在 provideInlineCompletionItems(document, position, context, token) 中：

   * 收集上下文（见下节）。
   * 做防抖（例如 300ms 内只触发一次远程请求）。
   * 调用 LLMClient.completeCode。
   * 将返回文本包装为 InlineCompletionItem，并指定 range 为当前光标位置之后。

Tab 接受补全是 VS Code 内置行为，不需要手动监听 Tab，只要使用 InlineCompletionItem 即可。

性能注意：如果用户持续输入，必须利用 CancellationToken，中途请求应被忽略，否则会出现“旧补全覆盖新状态”的问题。

五、上下文范围设计（核心能力）

你需要支持两类上下文：

1. 当前文件上下文
   从当前光标向前截取 N 行或 N tokens，例如前 200 行或 8k 字符。避免发送整文件，控制 token 成本和延迟。

2. 项目级上下文
   当用户设置为“项目路径下全部文件”时，不要真的把所有文件发给模型。应采用“相关文件检索”策略：

   * 只读取同目录或 import/require 引用的文件。
   * 限制文件总字节数上限。
   * 可缓存文件摘要（首次打开时生成）。

建议 contextCollector 输出统一结构：

{
language,
filePath,
prefixCode,   // 光标前
suffixCode,   // 光标后（用于补全质量提升）
relatedSnippets[]
}

PromptBuilder 根据不同任务（补全 / 文档改写）组装不同 Prompt 模板，而不是把拼接写死在各处。

六、文档改写与续写能力

通过注册 command 实现，例如：

* extension.rewriteSelection
* extension.continueWriting

命令触发后：

1. 获取选中文本或全文。
2. 弹出 quickPick 或 inputBox 让用户输入“改写为更正式/补充注释”等指令。
3. 调用 LLMClient.rewriteDocument。
4. 使用 TextEditor.edit 一次性替换选区内容。

注意：编辑操作必须是原子性的，避免多次插入导致光标错乱。

七、配置项设计规范

在 package.json 的 contributes.configuration 中定义：

* apiUrl: string
* apiKey: string (secretStorage 更优，不明文存 settings.json)
* modelName: string
* context.scope: enum [currentFile, project]
* context.maxLines: number

settings.ts 统一封装读取逻辑，并监听 onDidChangeConfiguration，做到运行时生效。

八、代码规范

TypeScript 严格模式开启，noImplicitAny 必须为 true。所有对外接口和数据结构必须显式声明类型，不允许 any 透传。

异步代码统一使用 async/await，不使用裸 Promise 链式写法。所有网络请求必须设置超时和异常捕获。

日志不得散落在业务代码中，只能通过统一 logger 工具输出到 OutputChannel。

不得在补全主路径做同步阻塞 IO，例如 fs.readFileSync 扫描整个项目，应缓存或异步分批处理。

九、性能与稳定性规范

补全请求是高频路径，必须有三道保护：

1. 防抖（减少请求量）。
2. 并发控制（同一文档只保留最新一次请求）。
3. 长文本裁剪（强制上限）。

任何模型超时或报错时，不得弹出阻塞式错误框，只在状态栏或 OutputChannel 提示，避免打断编码流。

十、可扩展性设计

后续若支持：

* 多模型切换
* RAG（本地向量检索）
* 代码库级记忆

都应只扩展 contextCollector 和 llmClient，不改 VS Code 集成层。所有 Prompt 模板集中在 promptBuilder，避免“魔法字符串”散落。

通过上述分层与规范，该插件既能实现 Copilot 风格实时补全，也能扩展为完整的本地大模型编程助手，同时保持良好的可维护性和性能边界。
